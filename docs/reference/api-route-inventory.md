### Service-Exported Routers (HTTP logic in wrong layer)

These four routers are imported from `src/backend/services/` rather than `src/backend/routes/`. Mixing HTTP route logic into service modules violates the layering convention (services should be pure business logic, not aware of HTTP context). They are candidates for extraction into proper route files during a future refactor:

1. `/api/outgoing-webhooks` — `src/backend/services/outgoingWebhooks.js`
2. `/api/email-marketing` — `src/backend/services/emailMarketing.js`
3. `/api/mfa` — `src/backend/services/enhancedMFA.js`
4. `/api/audit` — `src/backend/services/auditLog.js`

### Inline Handlers in server.js (candidates for extraction)

Highest-risk / highest-complexity inlines (most lines of business logic):

1. `/api/health/platforms` — ~180 LOC. Five parallel DB queries, uptime bucketing, incident aggregation, 30-second in-process cache. Should be extracted to `src/backend/routes/healthPlatforms.js`.
2. `/api/admin/affiliate-applications` — Direct `dbQuery` calls inside server.js. Should move to a route file or be merged into `affiliateRouter`.
3. `/api/feature-flags` — Full CRUD for feature flag management (GET list, GET all admin, PUT update). Should move to `src/backend/routes/featureFlags.js`.
4. `/api/user-analytics` — Three endpoints for page-view tracking and session retrieval. Should move to `src/backend/routes/userAnalytics.js`.
5. `/api/admin/workers/uptime-probe/trigger` — Admin-only dynamic import + business logic. Should be a route under `adminRouter` or a dedicated `src/backend/routes/adminWorkers.js`.

Lower-complexity inlines that are acceptable to leave in server.js (single-purpose, <10 LOC each):

- `/api/csrf-token` — returns `ctx.csrfToken`; no business logic.
- `/api/docs` — static spec URL pointers only.
- `/api/health`, `/api/health/live`, `/api/health/ready`, `/api/health/detailed` — standard health probes; acceptable in server.js for operational visibility without an import.
- `/api/status` — environment-gated status snapshot.
- `/api/workers/health` — Redis heartbeat reader; borderline but small.
- `/api/csp-report` — CSP violation logger; must stay public/no-CSRF by browser spec.

### Conditional Mounts

- `/mock-oauth` (and `/api/mock-oauth/`) — `src/backend/routes/mock-oauth.js`. Imported and mounted only when `process.env.NODE_ENV !== 'production'`. Also handled explicitly in the `fetch()` body before the `apiRoutes` dispatch loop. Ensure this never leaks into production.

## 3. Do Not Touch

`src/backend/server.js` must **not** be restructured, split, or have routes reordered until this inventory is used to plan the extraction safely. The `_sortedApiRoutes` array is sorted at startup by prefix length (longest first) to ensure most-specific routes match first — any reordering or partial extraction that changes this sort behavior risks routing bugs.

Before any extraction:
1. Confirm the target route file does not already exist or conflict with an existing router.
2. Update the `apiRoutes` object in server.js to replace the inline function with the imported router.
3. Add the new route file to the test suite.
4. Run `bun test src/tests/auth.test.js src/tests/security.test.js` after any change to server.js.
