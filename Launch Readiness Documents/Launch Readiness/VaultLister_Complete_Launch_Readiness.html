What "Launch Ready" Actually Means

VaultLister is a cross-platform reseller management SaaS tool. A reseller manages their entire secondhand selling business from one place — photographing and describing an item once, publishing it to multiple platforms simultaneously, and having sold items auto-delisted everywhere else. Launch ready means all of the following are true simultaneously. None of these is optional.

Core Product Loop

The central workflow that everything else exists to support
  • Add an item with photos, description, title, price, condition
  • Publish it to multiple platforms simultaneously
  • Item appears correctly on each destination platform
  • When it sells on one platform, it auto-delists on all others
  • Order is visible in VaultLister with real shipping rates
  • A printable label is generated through a real carrier account

Soft Launch Threshold (Week 12)

Minimum viable for first real users — eBay + Poshmark only
  • All 28 critical issues resolved
  • eBay + Poshmark integrations confirmed working end-to-end
  • Session persistence works (refresh-safe, bookmark-safe)
  • Stripe billing live — Free and Pro tiers
  • Shipping rates and label generation functional
  • No fake/hardcoded data anywhere

Full Launch Threshold (Week 16–17)

All 9 build phases + all 9 checklist sections complete
  • All 8 platform integrations confirmed + tested
  • Closed beta (≥5 resellers) completed, critical feedback resolved
  • Cross-browser + mobile tested (Chrome, Safari, Firefox, Edge, iOS, Android)
  • WCAG 2.1 AA accessibility baseline met
  • Legal documents reviewed by counsel, jurisdiction specified
  • All 9 launch readiness checklist sections signed off

What It Is Not

Scope boundaries
  • Not a marketplace — VaultLister doesn't host listings or handle transactions
  • Not responsible for platform policies — just integrating with them
  • Audit was Chrome-only — Safari/Firefox/mobile still need human testing
  • 18-session audit was one expert user — beta with real resellers is essential

The 10 Root Cause Patterns

Developer Handoff

These patterns, identified in the Developer Handoff, are responsible for the bulk of the ~416 issues. Fixing Pattern 1 resolves ~45 issues with one script change. Fixing Patterns 1–3 and 6–8 covers the majority of critical and significant issues.

P0 BLOCKER~45 ISSUES

Pattern 1 — handlers/modals undefined on hash-navigated pages

The JS initialisation entry point (handlers, modals, richTextEditor, focusMode, sessionMonitor, notificationCenter) is only called from the router's navigate() callback — not from DOMContentLoaded. Every page reached via direct URL (#community, #account, etc.) fails to initialise these objects silently. Fix: move all initialisation unconditionally to DOMContentLoaded. Add startup assertion: if (typeof handlers === "undefined") console.error("handlers not initialised").

P0 BLOCKERCRITICAL

Pattern 2 — Silent hash aliasing (8 routes)

#crosslist→#listings, #templates→#listings, #transactions→#financials, #about→#help-support, #notifications→#financials, #orders→#orders-sales→404, #billing→dead, #upgrade→dead. Users land silently on unrelated pages with no explanation. Fix: build the intended page or add a redirect with an explanatory message.

P2

Pattern 3 — Raw slug breadcrumbs (7+ pages)

"account", "plans-billing", "inventory-import", "Manage", "support-articles", "scanner > 404" rendered as breadcrumbs verbatim. Fix: humanise() utility in the router layer.

P1

Pattern 4 — Navigation black holes (8 functional pages with no sidebar entry)

#account, #plans-billing, #receipt-parser, #reports, #community, #roadmap, #inventory-import, #calendar all exist but have no navigation entry points. #plans-billing is the most urgent — it is the only upgrade page and is completely hidden. Direct revenue impact.

P1

Pattern 5 — Upgrade funnel broken end-to-end

#upgrade is dead, #plans is dead, #billing is dead, Settings/Billing tab is empty, "Manage Subscription" routes to #settings not #plans-billing, all three upgrade buttons on #plans-billing call undefined handlers. No user can currently upgrade to paid. Fix in order: (1) T1.1 handlers init, (2) fix Manage Subscription href, (3) add sidebar CTA, (4) verify #plans-billing loads via direct hash.

P0 BLOCKERLEGAL RISK

Pattern 6 — Hardcoded fake financial and market data

Financials: fabricated $29,200 revenue, $18,500 gross sales, $2,840 expenses, fake bank transactions. Analytics/Predictions: invented market trends shown as personalised predictions. Analytics/Market Intel: fake live competitor feed. Analytics/Sourcing: 4 fake supplier profiles with real-looking 555-prefix phone numbers (a user could call them). Settings/Data: fake login history including a "suspicious failed login" with real-looking IPs — a security-alarm-triggering event. Settings/Integrations: hardcoded "Connected" for eBay/Mercari/Whatnot when 0 shops are connected. Remove all of this before any user-facing launch.

P2

Pattern 7 — 0÷0 stats displayed as meaningful metrics

100% sales funnel conversion with 0 data. 100% Listing Health with 0 listings. 0% Completion Rate with 0 orders. 0% Acceptance Rate with 0 offers. All KPIs labelled "Poor" on a zero-data account. $5K Analytics goal vs $2K Dashboard goal (contradictory hardcoded values). Fix: global rule — any percentage stat with a zero denominator shows N/A. Hide performance metrics until at least one data point exists.

P0 BLOCKER

Pattern 8 — Session destroyed on hash navigation or page refresh

No token persisted to localStorage after login. Navigating to any hash route logs the user out. Clear Cache has no warning it logs the user out. Fix: on successful login, write JWT/session token to localStorage. On DOMContentLoaded, read token, validate, restore session before rendering any page.

P2

Pattern 9 — Platform count inconsistency ("9+" advertised, reality varies by page)

Landing page says "9+ marketplaces". My Shops shows 6. Automations filter omits eBay and Etsy. Listings filter omits Whatnot and Etsy. Plans & Billing says "6 platforms". Platforms page shows Etsy as "Pending app approval". Fix: centralise platform list as a single constant array. Update all marketing copy to reflect the actual live (not pending) count.

P0 BLOCKER

Pattern 10 — Add Item workflow: multiple broken dependencies

richTextEditor undefined, Save as Draft calls undefined handlers, all image upload tabs call undefined handlers, AI Generate / Scan Barcode / Use Template tabs all call undefined modals, form submits but item is not verifiably saved, no character limits enforced, no auto-save, platform selection not validated. The most critical user-facing workflow is almost entirely non-functional.
